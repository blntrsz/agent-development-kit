# adk-pi-docker prompt endpoint

## Goal

Convert `packages/adk-pi-docker/src/index.ts` from a one-shot CLI prompt into a Bun/Effect HTTP server exposing `POST /prompt`. The endpoint should return immediately with `accepted` and run the agent prompt in the background.

## Files

- `packages/adk-pi-docker/src/index.ts`
- `packages/adk-pi-docker/src/Dockerfile`

## Plan

1. Replace the top-level `await agent.prompt("Hello!")` flow in `packages/adk-pi-docker/src/index.ts` with an Effect HTTP server built from `HttpRouter`, `HttpServerResponse`, `BunHttpServer.layer`, and `BunRuntime.runMain`.
2. Keep the current agent defaults in one place (`systemPrompt`, `getModel("opencode-go", "kimi-k2.5")`) so each request uses the same configuration without duplicating literals.
3. Add `POST /prompt` as the only route. Parse the request body as JSON and validate a minimal schema: `{ prompt: string }`.
4. On valid input, generate a `requestId`, start a background Effect/fiber for the actual prompt work, and immediately return `202 Accepted` with JSON like `{ "status": "accepted", "requestId": "..." }`.
5. In the background task, create a fresh `Agent` per request instead of sharing one module-level instance. This avoids shared conversation state and mixed subscriptions across concurrent requests.
6. Subscribe to `message_update` / `text_delta` events inside that per-request task and write structured logs tagged with `requestId` instead of raw untagged stdout chunks.
7. Add small in-process concurrency control around the prompt runner (`Semaphore(1)` or another tiny cap) so the server can accept requests immediately while preventing prompt runs from overlapping uncontrollably.
8. Add explicit started/completed/failed logging around the background run so failures stay visible without breaking the HTTP response path.
9. Update `packages/adk-pi-docker/src/Dockerfile` to start the real entrypoint (`bun run src/index.ts`) so the container launches the new server.

## API Contract

Request:

```http
POST /prompt
Content-Type: application/json
```

```json
{
  "prompt": "Hello!"
}
```

Success response:

- `202 Accepted`

```json
{
  "status": "accepted",
  "requestId": "req_123"
}
```

Validation failure:

- `400 Bad Request`

```json
{
  "error": "Invalid request body"
}
```

## Verification

1. Run locally:

```bash
bun run packages/adk-pi-docker/src/index.ts
```

2. Hit the endpoint:

```bash
curl -i -X POST http://localhost:3000/prompt \
  -H 'content-type: application/json' \
  -d '{"prompt":"Hello!"}'
```

Expect `202 Accepted` and a JSON body with `status: accepted`.

3. Confirm it does not wait for completion:

```bash
time curl -s -X POST http://localhost:3000/prompt \
  -H 'content-type: application/json' \
  -d '{"prompt":"Write 200 words"}'
```

Expect curl to return immediately while server logs continue afterward.

4. Smoke-test multiple requests:

```bash
printf '%s\n' one two | xargs -I{} -P2 curl -s -X POST http://localhost:3000/prompt \
  -H 'content-type: application/json' \
  -d "{\"prompt\":\"{}\"}"
```

Expect both requests to be accepted and logs to stay attributable by `requestId`.
